The Handler interface

Everything in Go's HTTP server builds on a single interface with one method.

The contract

Go's http package has one rule: if your type has this method, it can handle HTTP requests. There is no implements keyword — if the method exists with the exact signature, your type satisfies the interface automatically at compile time.

type Handler interface {
    ServeHTTP(w http.ResponseWriter, r *http.Request)
}
// That's it. The entire http.Handler interface.
💡 This is called implicit interface satisfaction. No declaration needed — the compiler checks the method signature at compile time.

Notice that ServeHTTP returns nothing. The response is written into w like a stream — you call w.Write(), w.WriteHeader(), and w.Header().Set() to build the response. The function just ends when you're done.

Three ways to create a handler

Option A — struct + ServeHTTP
type MyHandler struct{
    service *Service
}

func (h *MyHandler) ServeHTTP(
    w http.ResponseWriter,
    r *http.Request) {
    w.Write([]byte("hello"))
}

// h is an http.Handler — use http.Handle
http.Handle("/path", &MyHandler{})
Option B — plain function
func myHandler(
    w http.ResponseWriter,
    r *http.Request) {
    w.Write([]byte("hello"))
}

// pass the function directly
// HandleFunc wraps it internally
http.HandleFunc("/path", myHandler)

Your pattern — Option C A method on a struct (not named ServeHTTP). Works fine with HandleFunc.

func (ctrl *WeatherController) GetWeatherHandler(
    w http.ResponseWriter, r *http.Request) { ... }

// ctrl.GetWeatherHandler is a "method value" — same signature as a plain func
http.HandleFunc("/weather", ctrl.GetWeatherHandler)

http.HandlerFunc — the converter type

A plain function with signature func(w, r) is NOT an http.Handler because it has no ServeHTTP method. http.HandlerFunc wraps it and adds the method. Its only job is conversion.

// From Go's standard library — this is all it is:
type HandlerFunc func(ResponseWriter, *Request)

func (f HandlerFunc) ServeHTTP(w ResponseWriter, r *Request) {
    f(w, r) // just calls itself
}

// Usage: convert a plain func into an http.Handler
var h http.Handler = http.HandlerFunc(myFunc) // ✓ now satisfies interface
🧩 You'll use http.HandlerFunc constantly inside middleware — it's how you return a new http.Handler from an anonymous function without defining a struct.

Handle vs HandleFunc — when to use which

FunctionAcceptsUsed when
http.Handle http.Handler interface Struct already has ServeHTTP, or you've wrapped with HandlerFunc
http.HandleFunc func(w, r) Plain functions or method values like ctrl.GetWeather

Internally, http.HandleFunc just calls http.Handle(path, HandlerFunc(fn)) — the conversion happens automatically.

Middleware chains

Functions that wrap handlers — running code before and after your actual logic.

What middleware is — one sentence

A function that takes an http.Handler, does something, and returns a new http.Handler.

func SomeMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {

        // ① runs on the way IN (before your handler)

        next.ServeHTTP(w, r) // ② passes control inward

        // ③ runs on the way OUT (after handler returns)
    })
}
💡 next.ServeHTTP(w, r) is just a method call on a variable. No magic. next holds a reference to whatever handler is "inside" this one.

You build this chain once at startup, wrapping inside → out:

weatherHandler := &WeatherHandler{}

http.Handle("/weather", logged)
// logged → rateLimited → weatherHandler
// each "next" variable holds a reference to the inner handler

The mutex-across-next trap

❌ Wrong — server becomes single-threaded
rl.mutex.Lock()
defer rl.mutex.Unlock()
// defer runs when func returns
// = lock held through ALL of this:
next.ServeHTTP(w, r) ← every downstream call
// DB queries, Redis, everything
// All other requests BLOCKED
✓ Right — lock released before next
func (rl *RateLimiter) isAllowed(ip string) bool {
    rl.mutex.Lock()
    defer rl.mutex.Unlock() ← scoped to THIS func
    // map work here
    return allowed
}
// then in middleware:
if !rl.isAllowed(ip) { ... }
next.ServeHTTP(w, r) ← no lock held ✓
⚠️ Extracting map logic into a helper function is the clean fix. defer inside the helper unlocks when it returns, not when the middleware function returns.

sync.Mutex

How to share state safely across goroutines — and why you must.

Why maps crash without a mutex

Your HTTP server creates a new goroutine for every single request. If two goroutines read and write the same map simultaneously, Go's runtime detects this and panics:

💥 fatal error: concurrent map read and map write — this is not just a bug, it's a runtime crash. Run with go run -race to catch it before production. 
// Two goroutines, same moment, same map:
// goroutine 1: count := rl.cache_map["1.2.3.4"]  ← READ
// goroutine 2: rl.cache_map["1.2.3.4"] = entry   ← WRITE
// Result: CRASH

// With mutex — only one goroutine at a time:
rl.mutex.Lock()             // all others wait here
count := rl.cache_map[ip]   // safe read
rl.cache_map[ip] = newEntry // safe write
rl.mutex.Unlock()           // next goroutine can enter

sync.Mutex — no initialization needed

type RateLimiter struct {
    mutex     sync.Mutex            // zero value = unlocked — just declare it
    limit     int
    cache_map map[string]userRateLimit
}

// No sync.NewMutex() needed — unlike channels or maps
// sync.Mutex{} is ready to use immediately
💡 A sync.Mutex must never be copied after first use. Pass your struct by pointer (*RateLimiter) not by value — otherwise each copy gets its own lock and protects nothing.

sync.Mutex vs sync.Map — chose correctly

sync.Mutex + mapsync.Map
Optimized for High write contention — same keys written constantly Write-once read-many, or keys accessed by disjoint goroutines
Rate limiter fit ✓ Correct — every request reads AND writes the same hot keys ✗ Wrong — optimized for the opposite access pattern
Internal mechanism Simple lock — predictable, low overhead Copy-on-write — adds overhead under write contention
Verdict For a rate limiter: faster AND simpler Would be slower here, not faster

Fixed window — what you implemented

type userRateLimit struct {
    count int
    time  int64  // unix timestamp of window start
}

// On each request:
if time.Now().Unix()-userRate.time > 10 {
    // window expired — reset
    rl.cache_map[ip] = userRateLimit{count: 0, time: time.Now().Unix()}
}
userRate = rl.cache_map[ip]        // re-read AFTER potential reset
if userRate.count >= rl.limit { /* block */ }
rl.cache_map[ip] = userRateLimit{
    count: rl.cache_map[ip].count + 1,
    time:  rl.cache_map[ip].time,   // preserve window start time
}
⚠️ Known limitation: boundary burst. A user can send (limit-1) requests at t=9s, window resets at t=10s, then (limit) more — getting nearly 2× the intended limit across a 2-second window. Redis sliding window fixes this.

Redis client & commands

Connecting to Redis, connection pooling, and the commands you'll use.

Creating the client with connection pooling

rdb := redis.NewClient(&redis.Options{
    Addr:         "redis:6379",  // docker service name, not localhost
    Password:     "",            // no password in dev
    DB:           0,             // default database
    PoolSize:     100,           // max concurrent TCP connections
    MinIdleConns: 10,            // keep 10 warm — no cold-start latency
})
🔗 Creating a new TCP connection per request costs ~1ms. A connection pool keeps N connections alive and reuses them. Your goroutine borrows one, uses it, returns it — same concept as your http.Client transport config in the weather service.

Core commands — your rate limiter toolkit

Redis commandWhat it doesGo call
GET key Read a string value. Returns nil if missing. rdb.Get(ctx, key).Result()
SET key val Write a value. Optional TTL: SET key val EX 60 rdb.Set(ctx, key, val, 60*time.Second).Err()
INCR key Atomically increment integer by 1. Creates key at 1 if missing. rdb.Incr(ctx, key).Result()
EXPIRE key secs Set TTL. Key auto-deletes after N seconds. rdb.Expire(ctx, key, 60*time.Second).Err()
TTL key Seconds remaining until expiry. Use for Retry-After header. rdb.TTL(ctx, key).Result()
DEL key Delete key immediately. rdb.Del(ctx, key).Err()

Context timeouts — always use them

// In your middleware — derive from the request's context
ctx, cancel := context.WithTimeout(r.Context(), 50*time.Millisecond)
defer cancel()

result, err := rateLimitScript.Run(ctx, rrl.rdb, ...).Int()
// If Redis takes > 50ms: ctx is cancelled, err != nil, you fail open
⚠️ Use r.Context() as the parent, not context.Background(). If the client disconnects mid-request, r.Context() is cancelled automatically — no orphan Redis calls.

Fail open vs fail closed

if err != nil {
    fmt.Printf("Redis error: %v\n", err)

    // FAIL OPEN — let request through (availability > strict correctness)
    next.ServeHTTP(w, r)  ← your current choice — correct for most APIs
    return

    // FAIL CLOSED — block all requests when Redis is down
    // http.Error(w, "Service Unavailable", 503)
    // Use for: financial APIs, abuse-critical systems
}
✓ Fail open is the right call for most APIs. An unavailable rate limiter should not take down your service. Users get through, and Redis being down is a monitoring problem, not a user problem.

Lua scripts

Atomic multi-command operations — the right way to do INCR + EXPIRE without a race.

Why Lua scripts exist — the race condition

Redis is single-threaded, so any individual command is atomic. But when you need two commands together atomically, there's a race window between them:

Goroutine A: INCR → 1
→
Goroutine B: INCR → 1
→
Both see count=1, neither sets EXPIRE
→
Key lives forever!

A Lua script runs entirely inside Redis's engine as a single unit. No other command can interleave while it executes.

Your Lua script — annotated

var rateLimitLuaScript = redis.NewScript(`
-- KEYS[1] = "rate:127.0.0.1"  (the Redis key for this IP)
-- ARGV[1] = 5                  (the limit)
-- ARGV[2] = 60                 (window in seconds)

local current = redis.call("GET", KEYS[1])

-- if key exists AND count >= limit: block
if current and tonumber(current) >= tonumber(ARGV[1]) then
    return 0    -- blocked

else
    current = redis.call("INCR", KEYS[1])  -- atomic increment

    if tonumber(current) == 1 then
        -- first request in this window — set the expiry
        redis.call("EXPIRE", KEYS[1], ARGV[2])
    end

    return 1    -- allowed
end
`)

KEYS vs ARGV — the critical distinction

KEYS[]ARGV[]
Contains Redis key names Extra data (limits, values, config)
Why separated Redis uses KEYS for cluster routing — tells it which shard holds the data Pure data, not used for routing
Index 1-based: KEYS[1], KEYS[2] 1-based: ARGV[1], ARGV[2]
Your script KEYS[1] = "rate:127.0.0.1" ARGV[1] = 5, ARGV[2] = 60
🚨 Lua arrays are 1-indexed, not 0-indexed like Go slices. KEYS[1] is the first key, not KEYS[0]. This trips everyone up the first time.

Calling it from Go

// .Run() signature: ctx, client, KEYS slice, then ARGV variadic args
allowed, err := rateLimitScript.Run(
    ctx,
    rrl.rdb,
    []string{key},  // KEYS — must be a []string
    limit,          // ARGV[1]
    window,         // ARGV[2]
).Int()           // convert redis.Cmd result to int

if allowed == 0 {
    w.Header().Set("Retry-After", "60")
    http.Error(w, "Too Many Requests", http.StatusTooManyRequests)
    return
}
⚡ redis.NewScript() uses EVALSHA under the hood. The first call sends the full script; Redis stores it by its SHA hash. All subsequent calls send only the hash — much less data over the wire.

Two-tier design

Why you need both local memory and Redis, and how Kubernetes breaks purely local state.

The fundamental trade-off

Tier 1 — Local memoryTier 2 — Redis
Speed Nanoseconds — no network hop ~1ms — network round trip
Correctness Wrong in multi-pod systems — each pod has its own counter Correct — single source of truth across all pods
Role Local bouncer — blocks obvious abuse instantly Global ledger — enforces real limits
Threshold 5× your real limit — catches bots only Actual limit (e.g. 100 req/min)

The Kubernetes consistency problem

3 pods, purely local state, limit = 100 req/min. Load balancer distributes traffic evenly:

Pod A: sees 34 req ✓
+
Pod B: sees 33 req ✓
+
Pod C: sees 33 req ✓
→
Actual total: 100 req — limit broken

Each pod thinks it's under limit. Your limit of 100 becomes effectively 300 with 3 pods. Redis solves this because all pods share one counter.

💡 Tier 1 does NOT try to enforce the real limit. It only stops clearly abusive traffic — e.g. 500 req/min on a single pod (clearly a bot). Normal users always reach Redis for the authoritative count.

Request flow through both tiers

func (rl *TwoTierLimiter) isAllowed(ctx context.Context, ip string) bool {

    // Tier 1: local check — no network, nanoseconds
    // threshold = 5x real limit — only blocks bots
    if !rl.localCache.Allow(ip, 500) {
        return false  // clearly abusive — reject locally, Redis never touched
    }

    // Tier 2: Redis check — global truth across all pods
    return rl.redis.Allow(ctx, ip, 100)  // real limit
}

Fixed window boundary burst — know what you have

Your window is anchored to the first request's timestamp, not to a rolling clock. This allows bursting at the boundary:

1

t = 8s — user sends 4 requests

Count = 4, under limit of 5. Window was started at t=0.

2

t = 9s — user sends 1 more request

Count = 5, limit reached. User is blocked.

3

t = 10s — window resets (diff > 10s)

New window starts. Count = 0. User can send 5 more.

4

t = 10s — user sends 5 more requests

Count = 5. All allowed. But: 10 requests happened between t=8 and t=11 — a 3-second window. Your 10-second limit was effectively 10 req in 3 seconds.

📖 A Redis sliding window (using sorted sets with ZADD + ZCOUNT) fixes this by always counting the last 10 seconds from right now, not from when the window started. That's the next step after this project.

Docker Compose — the one concept that trips you up

# docker-compose.yml
services:
  app:
    build: .
    ports: ["8080:8080"]
    environment:
      REDIS_ADDR: redis:6379   # ← service name, not localhost!
    depends_on: [redis]

  redis:
    image: redis:7-alpine
    ports: ["6379:6379"]
⚠️ Inside Docker Compose, your Go app connects to Redis via the service name (redis), not localhost. Each container is its own network namespace — localhost inside the app container refers to the app itself, not Redis.
